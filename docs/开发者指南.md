# 开发者指南

autodo-engine 负责运行时框架本身。开发引擎功能时，应优先保持调度、协议、审计与事务注册的边界清晰。

当前约定：引擎仓内保留最小可运行事务与最小工具集，保障“仅安装 engine 可跑 demo”；业务扩展能力由 autodo-kit 提供。

## 1. 架构分层

- autodoengine/core：类型、枚举、错误
- autodoengine/flow_graph：图加载、模板、编译与运行期节点结构
- autodoengine/scheduling：单步主循环、候选动作、决策规则、路由守卫
- autodoengine/taskdb：任务、步骤、日志、决策、快照、审计视图
- autodoengine/utils：路径、配置、事务注册、表达式与运行辅助
- autodoengine/tools/atomic：原子工具能力实现
- autodoengine/tools/public：manifest、schema、协议、权限、审计、facade
- autodoengine/tools/adapters：CLI、Python、Agent、Runner 统一入口

## 2. 与 autodo-kit 的边界

1. 引擎仓内保留最小事务目录 `autodoengine/affairs`，用于独立运行与最小 demo。
2. 若已安装 autodo-kit，事务扫描优先使用 `autodokit/affairs`；未安装时回退到 `autodoengine/affairs`。
3. run_affair、build_registry 同时兼容 `autodokit.affairs.*` 与 `autodoengine.affairs.*` runner 模块路径。
4. list_tools/get_tool 优先桥接 `autodokit.tools`；未安装时回退到 `autodoengine.tools`。

## 3. 包与入口

- Python 包名为 autodoengine
- CLI 入口为 autodoengine.main
- 对外 API 入口为 autodoengine.api
- Runner 脚本入口为 `scripts/invoke_capability.py`

## 5. 多入口约束

1. CLI、Python、Agent、Runner 不直接承载业务逻辑，只调用 `public.facade.invoke_capability`。
2. capability 输入/输出必须在 manifest/schema 中声明。
3. capability 调用默认写入审计记录到 `.autodoengine/tool_audits/`。
4. 新增 capability 时，必须同步补齐 schema、lint 回归和文档。

## 6. 开发约束

1. 新增路径字段时，先接入 utils/path_tools 的统一绝对路径预处理。
2. 不在调度层硬编码事务仓目录，统一通过事务注册系统发现。
3. docs 中涉及 CLI、包名、模块路径时必须使用 autodoengine 口径。
4. 新 capability 优先落在 `atomic/*`，根 `tools/` 不再新增平铺业务文件。
