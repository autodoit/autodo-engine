# 开发者指南

autodo-engine 负责运行时框架本身。开发引擎功能时，应优先保持调度、协议、审计与事务注册的边界清晰，不把官方事务实现重新带回本仓库。

## 1. 架构分层

- autodoengine/core：类型、枚举、错误
- autodoengine/flow_graph：图加载、模板、编译与运行期节点结构
- autodoengine/scheduling：单步主循环、候选动作、决策规则、路由守卫
- autodoengine/taskdb：任务、步骤、日志、决策、快照、审计视图
- autodoengine/utils：路径、配置、事务注册、表达式与运行辅助

## 2. 与 autodo-kit 的边界

1. 官方事务源码不再放在本仓库。
2. 事务扫描入口通过已安装的 autodokit 包定位 affairs 目录。
3. run_affair、build_registry、节点模板加载会依赖 autodo-kit 提供的 affair.json 与 affair.py。
4. list_tools/get_tool 会按需桥接 autodokit.tools，而不是在引擎仓复制一份 tools。

## 3. 包与入口

- Python 包名为 autodoengine
- CLI 入口为 autodoengine.main
- 对外 API 入口为 autodoengine.api

## 4. 开发约束

1. 新增路径字段时，先接入 utils/path_tools 的统一绝对路径预处理。
2. 不在调度层硬编码事务仓目录，统一通过事务注册系统发现。
3. docs 中涉及 CLI、包名、模块路径时必须使用 autodoengine 口径。
